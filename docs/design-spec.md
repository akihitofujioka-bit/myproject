## 1. 目的・背景

### 1.1 解決したい課題

議会だより（広報紙）の編集作業では、現状こんな手間が発生している:

- 議員から **Word・Excel など形式バラバラ**で記事が提出される。
- それを担当者が手作業で**決められた紙面レイアウトに流し込む**。
- 記事本文の体裁直し、誤字修正などの**編集**を毎回行う。
- 提出された**写真の切り抜き・明るさ調整**などを別ソフトで行き来して処理する。
- **議員の掲載順**（会派順・議席番号順など）を決め、差し替えると番号や位置がずれて崩れる。

これらを 1 つのアプリ内で完結させ、**「取り込み → 編集 → 並べ替え → レイアウト → 出力」**を一気通貫で行えるようにする。

### 1.2 ゴール（このソフトでできるようになること）

1. 提出記事（Word / Excel 等）をドラッグ＆ドロップで取り込む。
2. 取り込んだ記事本文をアプリ内で編集する。
3. 写真・画像を簡単に編集する（切り抜き・回転・明るさ調整など）。
4. 議員の掲載順を画面上で並べ替えて決定する。
5. 決められたレイアウトテンプレートに自動で流し込む。
6. 完成紙面を **Word と PDF** で出力する。

## 2. 対象ユーザー・利用シーン

| 項目 | 内容 |
| --- | --- |
| 主な利用者 | 議会事務局の広報担当者（1〜数名） |
| ITスキル想定 | Word / Excel は使えるが、DTP・デザインソフトは不慣れ |
| 利用頻度 | 議会だより発行ごと（年4回〜毎月など自治体により異なる） |
| 利用端末 | 事務局の Windows PC が中心（Mac も考慮） |
| 利用形態 | 各PCにインストールして単独で利用（ローカル完結） |

> 設計上の最重要方針: **「DTPソフトを使えない人でも迷わず1冊作れる」** ことを最優先にする。専門用語・複雑な操作を避ける。

## 3. 全体方針・技術スタック

### 3.1 アプリ形態（確定）

- **Electron によるデスクトップアプリ**（Windows / Mac 対応）。
- インストール型。インターネット接続不要で動作（ローカル完結）。
- データは利用PC内に保存する。

### 3.2 技術スタック（提案）

| 領域 | 採用候補 | 理由 |
| --- | --- | --- |
| アプリ基盤 | Electron | Word/PDF出力・ローカルファイル操作・配布に適し、Win/Mac両対応 |
| UI | React + TypeScript | 画面が多く状態管理が複雑なため。型安全で保守しやすい |
| ビルド | Vite + electron-builder | 開発が速く、インストーラ作成まで対応 |
| Word(.docx) 読み込み | mammoth.js | .docx を HTML/テキストへ変換。書式もある程度保持 |
| Excel(.xlsx) 読み込み | SheetJS (xlsx) | 表データの読み込みに定番 |
| 本文エディタ | Tiptap（ProseMirror系） | 見出し・太字・ルビ等のリッチ編集に対応。拡張しやすい |
| 画像編集 | Cropper.js + Canvas処理 | 切り抜き・回転は実績多数。明るさ等はCanvasフィルタで実装 |
| レイアウト描画 | HTML/CSS（印刷用CSS, paged.js 併用） | テンプレートをHTML/CSSで表現し、段組・ページ送りを再現 |
| PDF出力 | Electron `webContents.printToPDF` | 表示中のレイアウトHTMLをそのままPDF化（レイアウト一致） |
| Word出力 | `docx`（npm）/ docxtemplater | テンプレートに本文・画像を差し込んで.docx生成 |
| ローカル保存 | プロジェクトフォルダ（JSON + 画像） | 中身が見え、バックアップ・受け渡しが容易 |

> 📌 要確認: 上記ライブラリ構成は提案。特に **Word出力の再現度**は要素技術検証（PoC）で確認したい（§11参照）。

### 3.3 設計の基本思想

- **データ（中身）とレイアウト（見た目）を分離**する。
  記事・議員・画像は「データ」、テンプレートは「見た目」。並べ替えや差し替えで崩れないようにする。
- **テンプレート駆動**。紙面の体裁はテンプレート定義で持ち、自治体ごとに差し替え可能にする。
- **オフライン・ローカル完結**。個人情報（議員情報・原稿）を外部に送らない。

## 4. 機能要件

### 4.1 記事の取り込み

| ID | 要件 |
| --- | --- |
| F-IMP-1 | Word(.docx) を取り込み、本文テキストと基本書式（見出し・段落・太字・箇条書き）を抽出する |
| F-IMP-2 | Excel(.xlsx) を取り込み、表データを記事内の表として取り込む |
| F-IMP-3 | ドラッグ＆ドロップ / ファイル選択の両方に対応 |
| F-IMP-4 | 取り込んだ記事を「どの議員の記事か」「どのコーナーか」に割り当てる |
| F-IMP-5 | Word内に貼り込まれた画像を抽出し、画像素材として取り込む |
| F-IMP-6 | 文字数カウントを表示（割り当て枠の上限に対する超過/不足が分かる） |

📌 要確認:
- 対象とする入力形式は **.docx / .xlsx** を必須、**.doc（旧形式）/ PDF / プレーンテキスト**は対応するか。
- 自治体で**提出フォーマット（テンプレート）を統一**できるか（できると取り込み精度が大幅に上がる）。

### 4.2 記事の編集

| ID | 要件 |
| --- | --- |
| F-EDIT-1 | 本文をリッチテキストで編集（見出し/段落/太字/下線/箇条書き/表） |
| F-EDIT-2 | ルビ（ふりがな）の付与 ※議会だよりで需要が高い |
| F-EDIT-3 | 禁則・全角半角・誤字の簡易チェック（候補表示） |
| F-EDIT-4 | 記事タイトル・小見出し・本文・キャプションを項目として持つ |
| F-EDIT-5 | 編集の取り消し / やり直し（Undo/Redo） |
| F-EDIT-6 | 文字数が枠を超える場合に視覚的に警告 |

📌 要確認: 縦書きへの対応有無（議会だよりは縦書き紙面も多い）。→ §11のPoC対象。

### 4.3 画像（写真）の簡単な編集

| ID | 要件 |
| --- | --- |
| F-IMG-1 | 切り抜き（トリミング）。指定アスペクト比でのトリミングも可能 |
| F-IMG-2 | 回転 / 左右反転 |
| F-IMG-3 | 明るさ・コントラスト・彩度の調整 |
| F-IMG-4 | 拡大縮小・解像度の確認（印刷に耐える解像度か警告） |
| F-IMG-5 | 編集は非破壊（元画像を保持し、調整値を保存。やり直し可能） |
| F-IMG-6 | キャプション（写真説明）の付与 |

> 「簡単な編集」に絞る。レタッチ・合成などの高度機能は範囲外とする。

### 4.4 議員の掲載順の決定

| ID | 要件 |
| --- | --- |
| F-ORD-1 | 議員名簿を登録/編集（氏名・会派・議席番号・ふりがな・顔写真） |
| F-ORD-2 | 掲載順を画面上のドラッグ＆ドロップで並べ替え |
| F-ORD-3 | 並べ替えプリセット（会派順 / 議席番号順 / 五十音順 / 手動）を選択 |
| F-ORD-4 | 並べ替え結果が紙面レイアウトに即時反映され、番号や位置が自動で振り直される |
| F-ORD-5 | 会派ごとのグルーピング表示・色分け |

### 4.5 レイアウト・テンプレート

| ID | 要件 |
| --- | --- |
| F-LAY-1 | 紙面サイズ（A4 / B5 等）・段組・余白・フォント・見出し体裁をテンプレートで定義 |
| F-LAY-2 | 記事を「枠（フレーム）」に割り当て、テンプレートに沿って自動流し込み |
| F-LAY-3 | ページ追加 / 削除、記事の枠移動 |
| F-LAY-4 | あふれ（オーバーフロー）警告。枠に収まらない場合に明示 |
| F-LAY-5 | 表紙・目次・定型ページ（議決結果一覧など）のテンプレート |
| F-LAY-6 | プレビュー（実際の出力に近い見た目）をリアルタイム表示 |

📌 要確認: 既存の議会だよりのレイアウトを **1〜2号分サンプル提供**いただけると、標準テンプレートの精度が上がる。

### 4.6 出力（Word / PDF）

| ID | 要件 |
| --- | --- |
| F-OUT-1 | 完成紙面を **PDF** 出力（印刷所入稿・庁内印刷向け） |
| F-OUT-2 | 完成紙面を **Word(.docx)** 出力（出力後の微修正・庁内回覧向け） |
| F-OUT-3 | ページ指定出力 / 全ページ出力 |
| F-OUT-4 | 画像は印刷解像度を保って埋め込み |

> 📌 要確認: **PDFとWordで「どこまで見た目を一致させるか」**。
> PDFは画面プレビューと一致しやすいが、Wordは編集可能形式ゆえに**完全一致は技術的に難しい**。
> 想定方針: 「PDF＝完成版」「Word＝編集できる近似版」と役割分担する（PoCで再現度を確認）。

### 4.7 プロジェクト保存・管理

| ID | 要件 |
| --- | --- |
| F-PRJ-1 | 「1号分」を1プロジェクトとして保存（記事・画像・議員・レイアウトを一括保存） |
| F-PRJ-2 | プロジェクトはフォルダ形式（中に JSON 設定 + 画像素材）で保存し、受け渡し可能 |
| F-PRJ-3 | 自動保存・バックアップ |
| F-PRJ-4 | 過去号を複製して次号のひな型にできる |

## 6. データモデル（論理）

プロジェクトは下記の構造で保持する（JSONで永続化）。

```
Project（1号分）
├─ meta            : 号数, 発行日, 自治体名, 紙面サイズ
├─ councilMembers[]: 議員（後述）
├─ articles[]      : 記事（後述）
├─ images[]        : 画像素材（後述）
├─ layout          : ページ・枠割り当て（後述）
└─ template        : 使用テンプレートの参照と上書き設定
```

### 6.1 CouncilMember（議員）

```
CouncilMember
├─ id            : 一意ID
├─ name          : 氏名
├─ nameKana      : ふりがな（五十音順用）
├─ faction       : 会派
├─ seatNumber    : 議席番号
├─ portraitImage : 顔写真（imageへの参照, 任意）
└─ order         : 掲載順（手動並べ替え時の確定値）
```

### 6.2 Article（記事）

```
Article
├─ id            : 一意ID
├─ memberId      : 紐づく議員（任意。会派記事・事務局記事はnull可）
├─ sectionId     : 配置するコーナー/枠の種類
├─ title         : タイトル
├─ subtitle      : 小見出し
├─ body          : 本文（リッチテキスト/構造化データ）
├─ images[]      : 記事内画像（imageへの参照 + キャプション）
├─ sourceFile    : 取り込み元ファイル名（来歴）
└─ charCount     : 文字数（自動計算）
```

### 6.3 ImageAsset（画像素材）

```
ImageAsset
├─ id            : 一意ID
├─ originalPath  : 元画像（非破壊のため保持）
├─ edits         : { crop, rotate, flip, brightness, contrast, saturation }
├─ caption       : キャプション
└─ dpiWarning    : 印刷解像度が不足する場合のフラグ
```

### 6.4 Layout / Template

```
Template
├─ pageSize      : A4 / B5 ...
├─ margins       : 上下左右余白
├─ columns       : 段組数
├─ fonts         : 見出し/本文フォント・サイズ
└─ frames[]      : 枠定義（位置・サイズ・用途）

Layout
└─ pages[]
    └─ frameAssignments[] : どの枠にどの記事/画像を割り当てたか
```

> ポイント: 記事・議員・画像は「データ」、Templateは「見た目」。
> 並べ替えや差し替えをしても、Layoutが参照を付け替えるだけで紙面が崩れない構造にする。

## 8. アーキテクチャ構成

Electron の標準的な2プロセス構成を採用する。

```
┌────────────────────── Electron App ──────────────────────┐
│                                                           │
│  Renderer プロセス（React UI）                              │
│   ・各画面の表示と操作                                       │
│   ・本文エディタ / 画像編集 / プレビュー                      │
│        │  IPC（プロセス間通信）                              │
│        ▼                                                   │
│  Main プロセス（Node.js）                                   │
│   ・ファイル読み書き（プロジェクト保存・取り込み）              │
│   ・Word/Excel パース（mammoth / SheetJS）                  │
│   ・Word/PDF 出力（docx / printToPDF）                      │
│        │                                                   │
│        ▼                                                   │
│  ローカルファイルシステム（プロジェクトフォルダ）              │
└───────────────────────────────────────────────────────────┘
```

- **Renderer** は見た目と操作に専念。ファイルや重い変換処理は **Main** に IPC で依頼する。
- セキュリティ: `contextIsolation` 有効、`nodeIntegration` 無効、限定した IPC API のみ公開（preload）。

| # | 論点 | 内容 | 影響 |
| --- | --- | --- | --- |
| R-1 | Word出力の再現度 | .docxは編集可能形式のため、紙面プレビューと完全一致は難しい。役割分担（PDF=完成版 / Word=近似編集版）でよいか | 出力品質の期待値 |
| R-2 | 縦書き対応 | 議会だよりは縦書き紙面が多い。HTML/CSSの縦書き＋ページ送りは実装難度が上がる | レイアウト機能の範囲 |
| R-3 | 提出フォーマット統一 | 議員提出様式を統一できれば取り込み精度が大幅向上。統一できるか | 取り込みの品質 |
| R-4 | 既存レイアウトの提供 | 現行の議会だより1〜2号分を提供いただけると標準テンプレ精度が上がる | テンプレート品質 |
| R-5 | 入力形式の範囲 | .doc(旧形式)/PDF/テキスト等まで対応するか | 取り込みの対応範囲 |
| R-6 | 配布・更新 | 各PCへのインストール・アップデート方法（手動配布で良いか） | 運用負荷 |

## 12. 範囲外（このソフトでは扱わないこと）

- 高度な写真レタッチ・画像合成
- 印刷所ごとのトンボ・色校正など本格DTP入稿仕様（必要なら別途検討）
- Web版・スマホ版（今回はデスクトップのみ）
- クラウド共同編集・サーバー集約
