## 0. この文書について

本書は「議会だより作成支援ソフト」の設計・仕様をまとめたものです。
まず本書で**作るものの全体像と判断ポイント**を関係者間で合意し、その後に実装へ進むことを想定しています。

- 確定事項と未確定事項（要検討）を区別して記載します。
- 未確定事項は「📌 要確認」マークを付けています。実装着手前に決定が必要です。

## 1. 目的・背景

### 1.1 解決したい課題

議会だより（広報紙）の編集作業では、現状こんな手間が発生している:

- 議員から **Word・Excel など形式バラバラ**で記事が提出される。
- それを担当者が手作業で**決められた紙面レイアウトに流し込む**。
- 記事本文の体裁直し、誤字修正などの**編集**を毎回行う。
- 提出された**写真の切り抜き・明るさ調整**などを別ソフトで行き来して処理する。
- **議員の掲載順**（会派順・議席番号順など）を決め、差し替えると番号や位置がずれて崩れる。

これらを 1 つのアプリ内で完結させ、**「取り込み → 編集 → 並べ替え → レイアウト → 出力」**を一気通貫で行えるようにする。

### 1.2 ゴール（このソフトでできるようになること）

1. 提出記事（Word / Excel 等）をドラッグ＆ドロップで取り込む。
2. 取り込んだ記事本文をアプリ内で編集する。
3. 写真・画像を簡単に編集する（切り抜き・回転・明るさ調整など）。
4. 議員の掲載順を画面上で並べ替えて決定する。
5. 決められたレイアウトテンプレートに自動で流し込む。
6. 完成紙面を **Word と PDF** で出力する。

## 2. 対象ユーザー・利用シーン

| 項目 | 内容 |
| --- | --- |
| 主な利用者 | 議会事務局の広報担当者（1〜数名） |
| ITスキル想定 | Word / Excel は使えるが、DTP・デザインソフトは不慣れ |
| 利用頻度 | 議会だより発行ごと（年4回〜毎月など自治体により異なる） |
| 利用端末 | 事務局の Windows PC が中心（Mac も考慮） |
| 利用形態 | 各PCにインストールして単独で利用（ローカル完結） |

> 設計上の最重要方針: **「DTPソフトを使えない人でも迷わず1冊作れる」** ことを最優先にする。専門用語・複雑な操作を避ける。

## 3. 全体方針・技術スタック

### 3.1 アプリ形態（確定）

- **Electron によるデスクトップアプリ**（Windows / Mac 対応）。
- インストール型。インターネット接続不要で動作（ローカル完結）。
- データは利用PC内に保存する。

### 3.2 技術スタック（提案）

| 領域 | 採用候補 | 理由 |
| --- | --- | --- |
| アプリ基盤 | Electron | Word/PDF出力・ローカルファイル操作・配布に適し、Win/Mac両対応 |
| UI | React + TypeScript | 画面が多く状態管理が複雑なため。型安全で保守しやすい |
| ビルド | Vite + electron-builder | 開発が速く、インストーラ作成まで対応 |
| Word(.docx) 読み込み | mammoth.js | .docx を HTML/テキストへ変換。書式もある程度保持 |
| Excel(.xlsx) 読み込み | SheetJS (xlsx) | 表データの読み込みに定番 |
| 本文エディタ | Tiptap（ProseMirror系） | 見出し・太字・ルビ等のリッチ編集に対応。拡張しやすい |
| 画像編集 | Cropper.js + Canvas処理 | 切り抜き・回転は実績多数。明るさ等はCanvasフィルタで実装 |
| レイアウト描画 | HTML/CSS（`writing-mode: vertical-rl`, paged.js 併用） | テンプレートをHTML/CSSで表現。**縦書き**・段組・ページ送りを再現 |
| PDF出力 | Electron `webContents.printToPDF` | 表示中のレイアウトHTMLをそのままPDF化（レイアウト一致） |
| Word出力 | `docx`（npm）/ docxtemplater | テンプレートに本文・画像を差し込んで.docx生成。縦書きはセクションの文字方向（縦書き設定）で対応 |
| ローカル保存 | プロジェクトフォルダ（JSON + 画像） | 中身が見え、バックアップ・受け渡しが容易 |

> 📌 要確認: 上記ライブラリ構成は提案。特に **Word出力の再現度**は要素技術検証（PoC）で確認したい（§11参照）。

### 3.3 設計の基本思想

- **提出様式は統一しない前提で、アプリ内で"正規化"する**（最重要方針）。
  実態として提出物は **手書き原稿用紙・Word・Excel** が混在し、統一は困難。
  そこで取り込み時に**「原稿取り込みワークベンチ」**で、どの形式の提出物でも
  同じ記事データ（タイトル／本文／写真／キャプション）に整える（§4.1・§7参照）。
- **データ（中身）とレイアウト（見た目）を分離**する。
  記事・議員・画像は「データ」、テンプレートは「見た目」。並べ替えや差し替えで崩れないようにする。
- **テンプレート駆動**。紙面の体裁はテンプレート定義で持ち、自治体ごとに差し替え可能にする。
- **オフライン・ローカル完結**。個人情報（議員情報・原稿）を外部に送らない。
  → このため手書き原稿は**担当者による手入力（書き起こし）**で正規化する（自動OCR・クラウド送信は使わない）。

## 4. 機能要件

### 4.1 記事の取り込みと正規化（取り込みワークベンチ）

提出物は様式バラバラ（手書き・Word・Excel）である前提で、**1つの画面で同じ記事データに整える**。
これが本ソフトの「ワンクッション（正規化ステップ）」であり、以降の工程は形式を意識しない。

| ID | 要件 |
| --- | --- |
| F-IMP-1 | Word(.docx) を取り込み、本文テキストと基本書式（見出し・段落・太字・箇条書き）を抽出する |
| F-IMP-2 | Excel(.xlsx) を取り込み、表データを記事内の表として取り込む |
| F-IMP-3 | ドラッグ＆ドロップ / ファイル選択の両方に対応 |
| F-IMP-4 | 取り込んだ記事を「どの議員の記事か」「どのコーナーか」に割り当てる |
| F-IMP-5 | Word内に貼り込まれた画像を抽出し、画像素材として取り込む |
| F-IMP-6 | 文字数カウントを表示（割り当て枠の上限に対する超過/不足が分かる） |
| F-IMP-7 | **手書き原稿の書き起こし**: スキャン画像（PDF/JPEG/PNG）を画面に並べて表示し、担当者が本文を手入力できる |
| F-IMP-8 | **正規化ビュー**: 形式が何であれ、取り込み結果を「タイトル／本文／写真／キャプション」の項目に割り付けて確認・補正できる |
| F-IMP-9 | **入力形式の対応**: .docx / .xlsx を中心に、.doc（旧形式）/ PDF / プレーンテキスト、手書きはスキャン画像から手入力で対応する |
| F-IMP-10 | 取り込み元（手書き / Word / Excel 等）を来歴として記録する |

> ✅ 決定（R-3 見直し）: 提出様式の統一は**強制しない**。アプリ内の正規化で吸収する。
> ただし統一できる議員には**推奨様式（§15）**を任意で配布し、その分は取り込みを省力化する。
> ✅ 決定（手書き）: 自動OCRは使わず、**スキャン画像を見ながらの手入力（書き起こし）**で正規化する（オフライン原則の順守）。

### 4.2 記事の編集

| ID | 要件 |
| --- | --- |
| F-EDIT-1 | 本文をリッチテキストで編集（見出し/段落/太字/下線/箇条書き/表） |
| F-EDIT-2 | ルビ（ふりがな）の付与 ※議会だよりで需要が高い |
| F-EDIT-3 | 禁則・全角半角・誤字の簡易チェック（候補表示） |
| F-EDIT-4 | 記事タイトル・小見出し・本文・キャプションを項目として持つ |
| F-EDIT-5 | 編集の取り消し / やり直し（Undo/Redo） |
| F-EDIT-6 | 文字数が枠を超える場合に視覚的に警告 |

✅ 決定（R-2）: **縦書きに対応する（必須要件）**。議会だよりは縦書き紙面が中心のため。
本文編集は論理構造（見出し・段落・ルビ等）で扱い、縦書きの見た目はレイアウト/プレビュー側で適用する。

### 4.3 画像（写真）の簡単な編集

| ID | 要件 |
| --- | --- |
| F-IMG-1 | 切り抜き（トリミング）。指定アスペクト比でのトリミングも可能 |
| F-IMG-2 | 回転 / 左右反転 |
| F-IMG-3 | 明るさ・コントラスト・彩度の調整 |
| F-IMG-4 | 拡大縮小・解像度の確認（印刷に耐える解像度か警告） |
| F-IMG-5 | 編集は非破壊（元画像を保持し、調整値を保存。やり直し可能） |
| F-IMG-6 | キャプション（写真説明）の付与 |

> 「簡単な編集」に絞る。レタッチ・合成などの高度機能は範囲外とする。

### 4.4 議員の掲載順の決定

| ID | 要件 |
| --- | --- |
| F-ORD-1 | 議員名簿を登録/編集（氏名・会派・議席番号・ふりがな・顔写真） |
| F-ORD-2 | 掲載順を画面上のドラッグ＆ドロップで並べ替え |
| F-ORD-3 | 並べ替えプリセット（会派順 / 議席番号順 / 五十音順 / 手動）を選択 |
| F-ORD-4 | 並べ替え結果が紙面レイアウトに即時反映され、番号や位置が自動で振り直される |
| F-ORD-5 | 会派ごとのグルーピング表示・色分け |

### 4.5 レイアウト・テンプレート

| ID | 要件 |
| --- | --- |
| F-LAY-1 | 紙面サイズ（A4 / B5 等）・段組・余白・フォント・見出し体裁をテンプレートで定義 |
| F-LAY-2 | 記事を「枠（フレーム）」に割り当て、テンプレートに沿って自動流し込み |
| F-LAY-3 | ページ追加 / 削除、記事の枠移動 |
| F-LAY-4 | あふれ（オーバーフロー）警告。枠に収まらない場合に明示 |
| F-LAY-5 | 表紙・目次・定型ページ（議決結果一覧など）のテンプレート |
| F-LAY-6 | プレビュー（実際の出力に近い見た目）をリアルタイム表示 |
| F-LAY-7 | **縦書き紙面に対応**（縦書き／横書きをテンプレートで選択。段組・ページ送りも縦書きに追従） |

📌 要確認（R-4）: 既存の議会だよりのレイアウトを **1〜2号分サンプル提供**いただけると、標準テンプレートの精度が上がる。
→ 後日、参考として提供いただける予定。提供後に標準テンプレートを具体化する。

### 4.6 出力（Word / PDF）

| ID | 要件 |
| --- | --- |
| F-OUT-1 | 完成紙面を **PDF** 出力（印刷所入稿・庁内印刷向け） |
| F-OUT-2 | 完成紙面を **Word(.docx)** 出力（出力後の微修正・庁内回覧向け） |
| F-OUT-3 | ページ指定出力 / 全ページ出力 |
| F-OUT-4 | 画像は印刷解像度を保って埋め込み |

> ✅ 決定（R-1）: **役割分担で確定**。
> **「PDF＝完成版（画面プレビューと一致）」「Word＝出力後に微修正できる近似編集版」** とする。
> Word は編集可能形式ゆえ紙面プレビューとの完全一致は求めない。縦書き紙面の Word 出力再現度は P0（PoC）で確認する。

### 4.7 プロジェクト保存・管理

| ID | 要件 |
| --- | --- |
| F-PRJ-1 | 「1号分」を1プロジェクトとして保存（記事・画像・議員・レイアウトを一括保存） |
| F-PRJ-2 | プロジェクトはフォルダ形式（中に JSON 設定 + 画像素材）で保存し、受け渡し可能 |
| F-PRJ-3 | 自動保存・バックアップ |
| F-PRJ-4 | 過去号を複製して次号のひな型にできる |

## 6. データモデル（論理）

プロジェクトは下記の構造で保持する（JSONで永続化）。

```
Project（1号分）
├─ meta            : 号数, 発行日, 自治体名, 紙面サイズ
├─ councilMembers[]: 議員（後述）
├─ articles[]      : 記事（後述）
├─ images[]        : 画像素材（後述）
├─ layout          : ページ・枠割り当て（後述）
└─ template        : 使用テンプレートの参照と上書き設定
```

### 6.1 CouncilMember（議員）

```
CouncilMember
├─ id            : 一意ID
├─ name          : 氏名
├─ nameKana      : ふりがな（五十音順用）
├─ faction       : 会派
├─ seatNumber    : 議席番号
├─ portraitImage : 顔写真（imageへの参照, 任意）
└─ order         : 掲載順（手動並べ替え時の確定値）
```

### 6.2 Article（記事）

```
Article
├─ id            : 一意ID
├─ memberId      : 紐づく議員（任意。会派記事・事務局記事はnull可）
├─ sectionId     : 配置するコーナー/枠の種類
├─ title         : タイトル
├─ subtitle      : 小見出し
├─ body          : 本文（リッチテキスト/構造化データ）
├─ images[]      : 記事内画像（imageへの参照 + キャプション）
├─ source        : 取り込み元種別（手書き / Word / Excel / PDF / テキスト）＋ 元ファイル名（来歴）
├─ sourceScan    : 手書き時の参照スキャン画像（imageへの参照, 任意。書き起こし照合用）
└─ charCount     : 文字数（自動計算）
```

### 6.3 ImageAsset（画像素材）

```
ImageAsset
├─ id            : 一意ID
├─ originalPath  : 元画像（非破壊のため保持）
├─ edits         : { crop, rotate, flip, brightness, contrast, saturation }
├─ caption       : キャプション
└─ dpiWarning    : 印刷解像度が不足する場合のフラグ
```

### 6.4 Layout / Template

```
Template
├─ pageSize      : A4 / B5 ...
├─ margins       : 上下左右余白
├─ columns       : 段組数
├─ fonts         : 見出し/本文フォント・サイズ
└─ frames[]      : 枠定義（位置・サイズ・用途）

Layout
└─ pages[]
    └─ frameAssignments[] : どの枠にどの記事/画像を割り当てたか
```

> ポイント: 記事・議員・画像は「データ」、Templateは「見た目」。
> 並べ替えや差し替えをしても、Layoutが参照を付け替えるだけで紙面が崩れない構造にする。

## 7. 画面設計

### 7.1 画面一覧

| 画面 | 役割 |
| --- | --- |
| ① ホーム / プロジェクト一覧 | 新規作成・過去号を開く・複製 |
| ② 取り込み・正規化ワークベンチ | Word/Excel/画像/手書きスキャンを取り込み、同じ記事データへ整える（**ワンクッション**）。手書きはスキャン画像を見ながら手入力 |
| ③ 議員管理画面 | 議員名簿の登録・編集・掲載順の並べ替え |
| ④ 記事編集画面 | 本文のリッチ編集・文字数チェック |
| ⑤ 画像編集画面 | 切り抜き・回転・明るさ調整 |
| ⑥ レイアウト/プレビュー画面 | 枠への割り当て・紙面プレビュー |
| ⑦ 出力画面 | PDF / Word 出力 |

### 7.2 基本の作業フロー

```
① プロジェクト作成
        │
        ▼
② 提出記事を取り込み・正規化（ワンクッション）
   ├ Word/Excel … 自動抽出 → 項目に割り付け
   ├ 手書き原稿 … スキャン画像を見ながら本文を手入力（書き起こし）
   └ どの形式も「タイトル／本文／写真／キャプション」へ統一
        │
        ▼
③ 議員名簿を整え、掲載順を決定（D&D並べ替え）
        │
        ▼
④ 記事本文を編集（誤字直し・体裁）
        │
        ▼
⑤ 写真を切り抜き・明るさ調整
        │
        ▼
⑥ レイアウトに自動流し込み → プレビューで確認・微調整
        │
        ▼
⑦ PDF / Word で出力
```

### 7.3 画面イメージ（取り込み・正規化ワークベンチ ＝ 手書き書き起こしの例）

```
┌──────────────────────────────────────────────────────┐
│ 取り込み・正規化ワークベンチ        提出物: 山田議員_原稿.pdf │
├──────────────────────────┬───────────────────────────┤
```

## 8. アーキテクチャ構成

Electron の標準的な2プロセス構成を採用する。

```
┌────────────────────── Electron App ──────────────────────┐
│                                                           │
│  Renderer プロセス（React UI）                              │
│   ・各画面の表示と操作                                       │
│   ・本文エディタ / 画像編集 / プレビュー                      │
│        │  IPC（プロセス間通信）                              │
│        ▼                                                   │
│  Main プロセス（Node.js）                                   │
│   ・ファイル読み書き（プロジェクト保存・取り込み）              │
│   ・Word/Excel パース（mammoth / SheetJS）                  │
│   ・Word/PDF 出力（docx / printToPDF）                      │
│        │                                                   │
│        ▼                                                   │
│  ローカルファイルシステム（プロジェクトフォルダ）              │
└───────────────────────────────────────────────────────────┘
```

- **Renderer** は見た目と操作に専念。ファイルや重い変換処理は **Main** に IPC で依頼する。
- セキュリティ: `contextIsolation` 有効、`nodeIntegration` 無効、限定した IPC API のみ公開（preload）。

## 15. 推奨提出様式（任意・省力化のため）

> 方針見直し（R-3）: 提出様式の**統一は強制しない**。手書き・Word・Excel 混在は
> §4.1「取り込み・正規化ワークベンチ」で吸収する。
> 本章の様式は、**協力いただける議員向けの"任意の推奨様式"**であり、
> 使ってもらえた分だけ取り込みの手間（特に項目割り付け）が減る、という位置づけ。
> 強制しないので、守られなくても正規化ステップで対応できる。

協力可能な議員には **専用 Word テンプレート（.docx）** を任意配布する。
中身（原稿・写真）だけを記入してもらい、**レイアウト・縦書きはアプリ側で担保**する。

### 15.1 なぜ（推奨するなら）Word テンプレートか

| 案 | メリット | デメリット | 判定 |
| --- | --- | --- | --- |
| **専用Wordテンプレート** | 議員がWordに慣れている／記入欄・スタイル固定で取り込み精度が高い／導入障壁が低い | 様式を守らない提出があり得る（記入例で緩和） | ✅ 採用 |
| Excelフォーム | セル単位で構造が明確 | 長文本文が書きにくく議員負担が増える | 表データ用に限定 |
| 専用入力アプリ/Webフォーム | 取り込み精度は最高 | 議員側の習得・運用負荷が重い | 不採用 |

### 15.2 テンプレートの記入欄（固定スタイル）

専用スタイル名を付け、取り込み時にスタイルで項目を判別する。

| 欄 | スタイル名（例） | 内容 | 制約 |
| --- | --- | --- | --- |
| 記事タイトル | `議会だより_タイトル` | 記事の見出し | 全角◯字以内 |
| 議員名・会派 | `議会だより_氏名` | 氏名／会派／議席番号 | 名簿と照合 |
| 本文 | `議会だより_本文` | 記事本文 | **全角◯字以内**（枠に合わせ後日確定） |
| 小見出し | `議会だより_小見出し` | 本文中の見出し | 任意 |
| 写真キャプション | `議会だより_キャプション` | 写真の説明文 | 写真ファイル名と対応 |

> 文字数上限は R-4（既存号）提供後、枠サイズに合わせて確定する。

### 15.3 写真の提出ルール

- **本文に貼り込まず、別ファイルで添付**（JPEG / PNG）。
- ファイル名で本文の写真位置と対応付ける（例: `山田太郎_写真1.jpg`）。
- 推奨解像度を明記（印刷に耐える解像度。目安: 長辺1500px以上／350dpi相当）。
- 解像度不足はアプリ側で警告（F-IMG-4）。

> 理由: 写真を分離するとアプリの画像編集機能（切り抜き・明るさ調整）と相性が良く、印刷解像度も保てる。

### 15.4 禁止・非推奨事項（テンプレート記入時）

- テキストボックス・図形による独自レイアウト
- 表組みでの無理な段組（表データは Excel 様式で別途提出）
- 装飾的な書式（色・特殊フォント等）。**レイアウトはアプリ側で行うため中身のみ記入**

### 15.5 Excel 様式の役割

Excel は **「議決結果一覧」など表形式データ専用**とする。記入セルを定めた様式を別途配布する。
